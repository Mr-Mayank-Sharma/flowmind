Chapter 0 — Executive Summary
FlowMind is a unified, production-grade AI platform that combines the capabilities of autonomous self-learning agents, multi-channel communication gateways, visual workflow orchestration, intelligent coding assistance, and enterprise-grade security into a single coherent product. It is designed to be run locally, on-premise, or in the cloud — with zero external API vendor lock-in.

The platform is built from first principles, drawing architectural inspiration from four battle-tested open-source systems:
    • Hermes Agent (NousResearch) — closed learning loop, skill creation, cron automation, subagent parallelism
    • OpenClaw — multi-channel gateway, local-first philosophy, real-time canvas, model failover
    • n8n — visual pipeline builder with triggers, conditions, parallel branches, and 400+ integrations
    • OpenCode — AI-powered code assistant with terminal TUI and deep editor integration

FlowMind is NOT a wrapper around third-party AI services. Every connector, every integration, every protocol handler is built and owned by the platform itself. The only external dependency is the LLM inference layer — and even that is made optional via Ollama local inference.

2.3 Data Flow Architecture
User Request Flow (Chat):
    1. User sends message via Web UI or CLI
    2. API Gateway authenticates JWT, checks rate limits, routes to Agent Orchestrator via tRPC
    3. Agent Orchestrator queries Context Engine (Qdrant RAG + FTS5 session search)
    4. Retrieved context is injected into system prompt using Context Engine templates
    5. LLM Router selects optimal provider (local Ollama first, cloud fallback)
    6. Agent receives streamed response, identifies tool calls
    7. MCP Executor runs real tool calls with OAuth token management
    8. Results are streamed back to client via SSE
    9. Memory consolidation job is queued in BullMQ for background processing
    10. Skill Engine evaluates if new skill should be created from this interaction

2.4 Event Bus & Queue Architecture
All async operations flow through BullMQ on Redis with dedicated queues per concern:
    • agent-tasks — LLM inference requests, tool execution jobs
    • memory-consolidation — session summarization, embedding generation
    • skill-improvement — automated skill refinement jobs
    • pipeline-execution — workflow DAG execution steps
    • trigger-events — cron and webhook trigger processing
    • channel-outbound — message delivery to gateway channels
    • billing-events — Stripe webhook processing, usage metering
    • notifications — system alerts, quota warnings, completion events

2.5 Security Architecture
    • All inter-service communication is mTLS in production (k8s)
    • JWT tokens are short-lived (15 min access, 7 day refresh)
    • SAML assertions validated with org-specific IdP metadata
    • All API keys encrypted at rest with AES-256 using per-org keys
    • RBAC enforced at middleware level, not just UI
    • Audit log events are append-only (immutable via PostgreSQL triggers)
    • MCP OAuth tokens stored in encrypted column, auto-rotated
    • Rate limiting per user/org/endpoint via Redis sliding window

Chapter 4 — Agent Runtime & Self-Learning Engine
4.1 Agent Orchestrator
The Agent Orchestrator is a Python asyncio service that manages the full lifecycle of an AI interaction — from receiving a user request to delivering a final streamed response.

4.1.1 Agent Orchestrator Architecture
    • AgentSession — stateful object per conversation, holds working memory and tool context
    • PromptBuilder — assembles system prompt from: base template + user profile + skill list + retrieved context + tool definitions
    • ToolDispatcher — routes tool calls to MCP Executor or built-in tools with retry logic
    • StreamManager — handles SSE streaming back to the API gateway
    • MemoryWriter — queues memory consolidation jobs after each interaction

4.1.2 Context Engine Injection
Before every LLM call, the Context Engine retrieves relevant context and injects it into the prompt:
    11. User message is embedded using the configured embedding model
    12. Qdrant nearest-neighbor search retrieves top-K relevant memories
    13. FTS5 SQLite index (synced from PostgreSQL) searches past session summaries
    14. Retrieved context is ranked by relevance score and recency
    15. Top results are formatted and injected into the system prompt as a structured block
    16. User modeling data (preferences, goals, communication style) is appended

# packages/context-engine/retrieval.py async def retrieve_context(user_id: str, query: str, limit: int = 10) -> ContextBlock:     embedding = await embed(query)          # Vector search in Qdrant     vector_results = await qdrant.search(         collection_name=f"memories_{user_id}",         query_vector=embedding,         limit=limit,         score_threshold=0.72     )          # FTS search in SQLite (session summaries)     fts_results = await fts_search(user_id, query, limit=5)          # Merge, deduplicate, rank     merged = rank_and_merge(vector_results, fts_results)          return ContextBlock(         memories=merged[:limit],         user_model=await get_user_model(user_id)     )

4.2 Self-Learning Skill Engine
Inspired by Hermes Agent, the Skill Engine is the core differentiator of FlowMind — it makes the agent permanently smarter over time.

4.2.1 Skill Creation Loop
    17. After completing a complex multi-step task, the agent evaluates if a new skill should be created
    18. If the task took more than N tool calls or required novel reasoning, skill creation is triggered
    19. Skill Synthesizer generates: name, description, trigger pattern, and executable code
    20. Generated skill is validated (syntax check, safety scan) then stored in the DB
    21. Skill is embedded and added to the retrieval index so future agents can find it
    22. Agent is nudged in subsequent sessions to use and improve the skill

4.2.2 Skill Self-Improvement
Every time a skill is used, its outcome is tracked. The improvement engine runs weekly:
    • Skills with < 70% success rate are queued for improvement
    • LLM analyzes failure cases from execution logs
    • Improved skill code is generated and diff-validated
    • A/B testing determines if the new version outperforms the old
    • New version is promoted if it wins, old version is archived

4.2.3 User Modeling (Honcho-compatible)
    • Communication style preferences (formal/casual, verbose/concise)
    • Expertise domains (programming languages, tools, topics)
    • Goal tracking (short-term tasks, long-term objectives)
    • Behavioral patterns (working hours, preferred output formats)
    • Feedback signals (explicit thumbs up/down, implicit corrections)

4.3 Self-Healing Retry with Exponential Backoff
Every LLM call and tool execution is wrapped in a resilient retry handler:

# packages/agent-runtime/resilience.py import asyncio, random from typing import TypeVar, Callable, Awaitable  T = TypeVar("T")  async def with_retry(     fn: Callable[[], Awaitable[T]],     max_attempts: int = 5,     base_delay: float = 1.0,     max_delay: float = 60.0,     jitter: bool = True,     retryable_errors: tuple = (RateLimitError, TimeoutError, ServiceUnavailableError) ) -> T:     attempt = 0     while attempt < max_attempts:         try:             return await fn()         except retryable_errors as e:             attempt += 1             if attempt >= max_attempts:                 raise             delay = min(base_delay * (2 ** (attempt - 1)), max_delay)             if jitter:                 delay += random.uniform(0, delay * 0.1)             await asyncio.sleep(delay)         except Exception as e:             # Non-retryable: log and re-raise immediately             logger.error(f"Non-retryable error: {e}")             raise  # Tool execution with self-healing class ToolExecutor:     async def execute(self, tool_call: ToolCall) -> ToolResult:         return await with_retry(             lambda: self._execute_internal(tool_call),             max_attempts=3,             base_delay=2.0         )

4.4 Cron Scheduler
FlowMind has a built-in cron system inspired by Hermes Agent, using BullMQ's repeat functionality:
    • Natural language scheduling: 'every weekday at 9am' parsed by LLM to cron expression
    • Standard cron syntax also supported: '0 9 * * 1-5'
    • Delivery to any connected channel (Telegram, Slack, Email, Webhook)
    • Missed-run recovery: if a job was scheduled during downtime, it runs immediately on startup
    • Admin UI to view, pause, edit, and delete scheduled jobs
    • Per-user cron quota enforcement (Free: 3, Pro: 20, Team: unlimited)

Chapter 5 — Real MCP Tool Execution Layer
5.1 MCP Protocol Implementation
FlowMind implements the Model Context Protocol (MCP) natively — not as a wrapper around a third-party service, but as a first-class protocol handler built into the platform.

5.1.1 MCP Executor Architecture
    • McpServer Registry — catalog of available MCP servers with capabilities, versions, and auth requirements
    • McpConnection Pool — persistent connections to active MCP servers (SSE and stdio transports)
    • McpToolRouter — maps tool names to server connections and validates schemas
    • McpResultParser — normalizes tool results into a standard internal format
    • McpAuditLogger — records every tool execution in the audit log

5.1.2 Real OAuth 2.0 Flows (GitHub, Slack, Google, Notion, Linear, Jira)
    23. User initiates OAuth from Settings → Connections page
    24. FlowMind redirects to provider's authorization URL with PKCE
    25. Provider redirects back to FlowMind callback endpoint
    26. Authorization code is exchanged for access + refresh tokens
    27. Tokens are encrypted with AES-256 and stored in McpToken table
    28. Token refresh is handled automatically before expiry
    29. Scopes are stored and displayed to user for transparency

7.3 Condition & Parallel Branch UX
The pipeline canvas uses React Flow for the node editor with custom node types:
    • Condition node: opens an inspector panel with field selector + operator + value. Output sockets are labeled TRUE and FALSE
    • Switch node: dynamic output sockets, one per case value plus a DEFAULT fallback
    • Parallel node: forks execution to multiple branches simultaneously. BullMQ jobs are created for each branch
    • Merge node: waits for all parallel branches to complete before continuing (barrier synchronization)
    • Inspector panel: click any node to open a right-side panel showing all configuration, including wiring visualization

7.4 Pipeline Execution Engine
// packages/workflow-engine/src/executor.ts export class PipelineExecutor {   async execute(pipelineId: string, input: Record<string, unknown>) {     const pipeline = await db.pipeline.findUnique({ where: { id: pipelineId } });     const run = await db.pipelineRun.create({ data: { pipelineId, status: "RUNNING" } });          const graph = new DAG(pipeline.graph);     const startNodes = graph.getStartNodes();          // Execute start nodes, then traverse     await Promise.all(startNodes.map(node => this.executeNode(node, input, run.id, graph)));   }    async executeNode(node: PipelineNode, input: any, runId: string, graph: DAG) {     const handler = NodeHandlerRegistry.get(node.type);          const result = await withRetry(       () => handler.execute(node.config, input),       { maxAttempts: node.config.retries ?? 3, baseDelay: 2000 }     );          await db.runLog.create({ data: { runId, nodeId: node.id, output: result } });          // Determine next nodes based on result (handles condition branching)     const nextNodes = graph.getNextNodes(node.id, result);          if (node.type === "parallel") {       await Promise.all(nextNodes.map(n => this.executeNode(n, result, runId, graph)));     } else {       for (const next of nextNodes) {         await this.executeNode(next, result, runId, graph);       }     }   } }

10.2 API Key Management
API keys are stored securely with the following design:
    • On creation: key is shown once in full, then only the last 4 characters are shown
    • Key is hashed with SHA-256 before storage — only the hash is in the DB
    • Key name and provider are stored in plaintext for UI display
    • Keys are tested on save — invalid keys are rejected with clear error messages
    • Usage tracking: last-used timestamp is updated on each API call
    • Key rotation: generate a new key, grace period for old key, then revoke
